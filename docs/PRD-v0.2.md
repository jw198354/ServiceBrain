## 2. 产品概述

### 2.1 产品背景

当前需设计一款面向电商客服咨询场景的 H5 智能客服机器人 Demo，用于完整展示智能客服产品在真实业务场景中的核心能力，包括自然对话、多轮理解、问题归类、参数追问、知识答疑、规则判断、工具执行与异常兜底。

在初稿基础上，当前阶段目标从"方向正确、链路完整"进一步升级为"**规则可执行、流程可验证、研发可实现、测试可覆盖**"。因此本稿重点解决以下问题：

1. 将原则性表述转化为功能约束与状态约束。 
2. 将主流程补足为主流程、分支流程、异常流程。 
3. 将 AI 能力表述落到具体产品处理策略。 
4. 将验收标准细化为可验证的验收场景。 
5. 将埋点指标定义为可落地的口径。 

### 2.2 产品定位

本产品定位为：

面向电商客服咨询场景、以 H5 聊天式交互为载体、强调"自然对话 + AI 理解 + 业务规则 + 工具调用 + 异常兜底"的智能客服机器人 Demo 产品。

本稿细化后，产品不再仅停留于"可演示能力说明"，而是进一步明确为：

- 具备最小可行的业务状态管理能力
- 具备核心消息类型和页面状态定义
- 具备主流程与异常流程的闭环设计
- 具备研发实现边界与测试覆盖边界
- 具备 MVP 排期优先级依据

### 2.3 建设目标

在初稿目标基础上，细化目标如下：

1. 建立可支持排期评估的功能边界与优先级体系。 
2. 建立适合聊天式产品的状态流转定义。 
3. 明确"AI 判断、规则判断、工具结果"三者的边界。 
4. 明确退款咨询、退款执行、规则解释、知识答疑四类链路的差异。 
5. 建立对研发、测试、算法均可消费的统一产品语言。 

### 2.4 目标用户

#### 2.4.1 终端用户
- 电商平台消费者
- 访问 H5 页面寻求帮助的售前或售后用户

#### 2.4.2 内部用户
- 产品评审人员
- 研发与测试团队
- AI / 算法团队
- 业务运营 / 客服运营团队
- 管理层 / 对外演示对象

### 2.5 核心问题定义

在初稿问题基础上，细化为以下具体设计挑战：

1. 用户表达是自然语言，系统处理必须结构化。 
2. 用户诉求往往隐含在问题描述中，不能依赖机械式双步骤收集。 
3. 退款相关语句高频混合"咨询 + 执行 + 情绪 + 解释诉求"。 
4. 知识问答不是简单 FAQ 命中，而需要经过客服化重组。 
5. 工具链路不能因 AI 错判或参数缺失产生误执行。 
6. 聊天式产品如果缺少页面状态与消息类型设计，易出现体验混乱。 

### 2.6 解决方案概览

在原解决方案基础上，补充产品细化层面：

- 用**页面状态机**保障用户从进入到会话初始化的体验连续性
- 用**消息类型定义**承载不同处理结果的展示差异
- 用**主流程 / 分支流程 / 异常流程**明确各场景处理分流
- 用**规则判断层**接管高风险节点
- 用**AI 处理策略**完成理解、追问、组织与解释
- 用**埋点和验收用例**反推实现与测试边界

### 2.7 产品范围

本期范围在原稿基础上保持不变，并进一步定义为：

#### P0 范围（MVP 必须）
- 匿名接入
- 用户名采集
- 会话创建
- WebSocket 消息通讯
- 首问
- 多轮对话
- 订单号补问
- 主意图归类
- 退款咨询 / 执行区分
- 知识答疑
- 规则解释
- 退款工具调用
- 工具失败 / 知识未命中兜底
- 工单引导
- 核心埋点

#### P1 范围（建议本期实现）
- 快捷问题区
- 会话恢复
- 消息状态展示
- 轻量结果卡片
- 用户连续模糊表达的选项式引导

#### P2 范围（后续增强）
- 历史会话管理
- 更细粒度规则配置后台
- 多轮主题切换优化
- 更丰富运营能力

### 2.8 非产品范围

本期不建设真正的"多渠道统一客服平台"，也不建设"完整人工客服系统"，Demo 中工单引导以闭环体验和信息传递为目标，不强求真实后端处理闭环。

## 3. 产品设计原则

### 3.1 设计原则说明

#### 原则一：低门槛接入
不引入复杂注册登录，仅以用户名完成最小身份初始化。

#### 原则二：自然对话优先
先让用户表达问题，再按需追问，不将聊天产品退化为多字段表单。

#### 原则三：AI 理解与业务规则并重
AI 负责理解与组织，规则负责约束与判定，工具负责执行。

#### 原则四：先解释，再执行
退款类场景必须先判断用户是在问规则、问资格，还是要求执行。

#### 原则五：受控式分流
意图树只做主意图归类，真正链路由上下文、槽位和规则共同决定。

#### 原则六：失败有兜底
失败不是中止，而是进入下一步可推进路径。

#### 原则七：Demo 闭环与产品化兼容
既满足演示，也避免写死为一次性脚本。

### 3.2 从产品原则到功能约束

#### 3.2.1 "自然对话优先" 的功能约束
1. 首轮问候中不得一次性要求用户提供用户名以外的多项业务字段。 
2. 用户首次业务表达后，系统优先尝试从原话中提取问题、诉求、订单信息。 
3. 追问遵循"最小必要原则"，每轮仅补一个关键缺失信息。 
4. 若当前问题无需订单号，不得主动追问订单号。 
5. 用户已明确表达诉求时，不得再次机械确认"你的诉求是什么"。 

#### 3.2.2 "失败有兜底" 的功能约束
1. 知识无命中时，不允许直接返回"未找到答案"作为终局。 
2. 工具失败时，不允许沉默、假成功或只暴露错误码。 
3. 意图不明时，需至少提供一次追问和一次场景提示。 
4. 用户不接受当前结果时，需有解释增强或工单引导。 

#### 3.2.3 "受控式分流" 的功能约束
1. 意图树输出不可直接等于最终动作。 
2. 工具调用前必须经过规则校验。 
3. 规则解释问题优先走"解释链路"，不直接触发执行链路。 
4. 退款咨询与退款执行必须区分，不允许因出现"退款"关键词即默认执行。 

### 3.3 对用户体验和业务价值的影响

- 用户侧：减少被盘问感，提升被理解感。 
- 平台侧：降低误执行、误回答、误承诺风险。 
- 研发侧：边界更清晰，状态更可控。 
- 测试侧：流程覆盖更可枚举，验收更可执行。

## 4. 用户与场景分析

### 4.1 用户角色定义

| 用户类型 | 特征 | 处理复杂度 |
|---|---|---|
| 首次进入用户 | 对产品不熟悉，需要低门槛启动 | 低 |
| 售前咨询用户 | 目标明确，多为规则或说明类 | 低 - 中 |
| 售后咨询用户 | 强依赖订单上下文 | 中 |
| 退款咨询用户 | 关注是否可退、时效、流程 | 中 |
| 退款执行用户 | 有明确动作诉求 | 中 - 高 |
| 解释型用户 | 更关注原因和依据 | 高 |
| 无法被直接解决用户 | 表达模糊、超边界、异常复杂 | 高 |

### 4.2 典型用户诉求

| 用户原话 | 问题识别 | 诉求识别 |
|---|---|---|
| 退款多久到账 | 退款时效规则 | 咨询 |
| 能不能退 | 退款资格 | 咨询 |
| 帮我退款 | 退款申请 | 执行 |
| 为什么不能退款 | 当前不可退原因 | 解释 |
| 订单还没到 | 物流/履约异常 | 咨询/处理 |
| 这个有问题 | 模糊问题 | 待追问 |

### 4.3 核心业务场景

| 场景 | 描述 | 优先级 |
|---|---|---|
| 首次进入 | 用户名输入、身份初始化、首问 | P0 |
| 售前规则答疑 | 运费、发货、服务说明 | P0 |
| 订单/物流咨询 | 订单状态、物流问题 | P0 |
| 退款咨询 | 能否退、如何退、多久到账 | P0 |
| 退款执行 | 发起退款申请 | P0 |
| 规则解释 | 为什么失败 / 为什么不能退 | P0 |
| 工单兜底 | 无法直接解决时兜底 | P0 |
| 快捷问题引导 | 降低启动门槛 | P1 |
| 历史会话恢复 | 提升连续性 | P1 |

### 4.4 售前与售后差异分析

| 维度 | 售前 | 售后 |
|---|---|---|
| 订单号要求 | 非必填 | 多数必填 |
| 追问策略 | 以场景澄清为主 | 以订单上下文补齐为主 |
| 规则风险 | 较低 | 高 |
| 结果形式 | 解答为主 | 解答 / 解释 / 执行为主 |
| 工单必要性 | 较低 | 较高 |

### 4.5 用户旅程分析

1. 未进入状态 
2. 首次进入待初始化状态 
3. 用户名弹窗状态 
4. 会话初始化中状态 
5. 已连接待首问状态 
6. 首问完成待输入状态 
7. 系统理解处理中状态 
8. 追问等待状态 
9. 结果返回状态 
10. 异常 / 工单兜底状态 

### 4.6 关键痛点分析

| 痛点类型 | 描述 |
|---|---|
| 体验痛点 | 盘问式交互、不知道下一步 |
| 理解痛点 | 用户一句话中同时包含多种信息 |
| 状态痛点 | 消息处理中、连接中断、工具调用时用户无感知 |
| 风险痛点 | 误执行退款、错误解释原因 |
| 兜底痛点 | 知识没命中或工具失败后无闭环 |

### 4.7 用户故事（新增）

#### 4.7.1 首次进入用户故事
- 作为一个第一次使用 H5 客服机器人的用户，我希望只输入一个简单用户名就可以开始咨询，以便快速进入问题解决流程。 
- 作为一个不想注册账号的用户，我希望不要在进入前填写太多资料，以便降低使用门槛。 

#### 4.7.2 售前咨询用户故事
- 作为一个准备下单的用户，我希望直接问发货、运费、售后规则等问题，以便快速判断是否购买。 
- 作为一个售前用户，我希望系统不要强制我要提供订单号，以便不打断我的咨询体验。 

#### 4.7.3 售后退款咨询用户故事
- 作为一个售后用户，我希望先知道这个订单能不能退、怎么退、多久到账，以便判断是否继续申请退款。 
- 作为一个售后用户，我希望机器人能结合我的订单情况解释原因，而不是只给我一段规则原文。 

#### 4.7.4 售后退款执行用户故事
- 作为一个已经决定退款的用户，我希望在提供必要信息后由机器人直接帮我发起退款，以便减少人工操作步骤。 
- 作为一个退款执行用户，我希望在不能退款时得到清晰原因和替代路径，以便知道下一步怎么办。 

#### 4.7.5 规则答疑用户故事
- 作为一个遇到规则限制的用户，我希望知道"为什么不行"，以便理解当前限制，而不是只被告知结果。 

#### 4.7.6 需要兜底的用户故事
- 作为一个表达不清或问题复杂的用户，我希望机器人在暂时处理不了时，仍然能帮助我整理问题并引导后续处理，以便不被中断在半路。

## 5. 产品价值分析

### 5.1 用户价值
- 启动成本低
- 对话自然
- 回复可理解
- 问题可推进

- 规则问答自动化
- 高风险动作受控自动化

- 降低无效人工接待

- 可走通标准成功链路

### 5.6 后续产品化价值
- 意图、槽位、规则、消息状态可直接演进为正式产品底座

## 6. 整体方案设计

### 6.1 整体产品方案概述

1. **接入层**：用户进入、匿名身份、用户名采集 
2. **连接层**：会话创建、WebSocket 建联、重连恢复 
3. **交互层**：消息流、输入框、状态提示、快捷问题 
4. **理解层**：主意图识别、问题/诉求联合识别、实体抽取 
5. **决策层**：槽位补齐、知识/规则/工具路径选择 
6. **执行层**：知识检索、资格判断、退款工具调用 
7. **回复层**：客服化话术生成、解释、建议、工单兜底 
8. **观测层**：埋点、日志、指标 

### 6.2 端到端用户链路

- 用户名提交成功 → user_init_status=done
- 会话创建成功 → session_status=active
- WebSocket 连接成功 → connection_status=connected
- 首问发送完成 → first_greeting_status=done
- 当前轮进入 AI 理解 → turn_status=understanding
- 等待用户补充 → slot_status=partial
- 工具处理中 → tool_status=processing
- 结果完成 → turn_status=completed
- 工单引导 → fallback_status=ticket_suggested

### 6.3 核心处理链路

1. 先判断是否为违规/越权问题 
2. 再判断是否已有足够信息直接处理 
3. 再判断是否需要补槽 
4. 再判断走知识/规则解释/执行工具 
5. 最后进入兜底 

### 6.4 主流程与兜底流程概览

| 类型 | 标准主流程 | 参数缺失分支 | 识别不清分支 | 执行失败分支 | 用户不接受分支 |
|---|---|---|---|---|---|
| 售前咨询 | 知识答疑 | 场景追问 | 轻量选项提示 | 不适用 | 推荐相关问题 |
| 售后咨询 | 补订单号 + 查询/解释 | 订单号补问 | 追问问题点 | 查询失败转工单 | 原因说明 + 工单 |
| 退款咨询 | 规则说明 | 补订单号（如需资格判断） | 追问场景 | 资格查询失败 | 解释 + 工单 |
| 退款执行 | 补参数 + 资格校验 + 工具调用 | 参数补问 | 确认是否真的要执行 | 工具失败 | 解释 + 工单 |
| 规则解释 | 规则 + 状态解释 | 补订单号/上下文 | 追问失败点 | 状态查询失败 | 保守解释 + 工单 |

### 6.5 功能模块关系说明

- AI 理解层不直接做业务授权 
- 规则判断层不负责自然语言回复 
- 工具调用层不负责最终用户沟通 
- 兜底层承接一切"无法直接完成闭环"的情况

## 7. 功能架构设计

### 7.1 功能模块总览

| 模块 | 职责 | 优先级 |
|---|---|---|
| 用户接入层 | 匿名进入、用户名采集 | P0 |
| 会话交互层 | 聊天 UI、消息展示、输入 | P0 |
| AI 理解与路由层 | 问题/诉求识别、意图归类、槽位识别 | P0 |
| 知识问答层 | 知识检索与回答组织 | P0 |
| 规则判断层 | 资格判断、触发条件判断 | P0 |
| 工具调用层 | 退款执行 | P0 |
| 兜底处理层 | 追问、解释、工单 | P0 |
| 会话与状态管理层 | 连接与上下文状态 | P0 |
| 运营配置层 | 首问、快捷问题、规则配置 | P1 |

### 7.2 页面状态设计（新增）

#### 7.2.1 页面状态枚举

- 若失败则进入 fallback_ticket 或 connection_error

## 8. 核心业务流程设计

> 本章在初稿基础上，统一使用"主流程 / 分支流程 / 异常流程"进行细化。

## 8.1 匿名进入与用户名采集流程

### 功能目标
完成最小身份初始化并进入可对话状态。

### 触发条件
用户首次访问 H5 页面，且本地不存在有效匿名身份。

### 前置条件
页面加载成功。

### 主流程
1. 页面加载完成。 
2. 系统检测无匿名身份。 
3. 弹出用户名输入弹窗。 
4. 用户输入合法用户名并确认。 
5. 系统创建匿名账号。 
6. 进入会话初始化流程。 

### 分支流程
- 若本地已有有效匿名身份：
 1. 直接跳过用户名弹窗。
 2. 进入会话恢复或新会话初始化。

### 异常流程
- 用户未输入：提示"请输入用户名后开始咨询"
- 用户名非法：提示修改，不关闭弹窗
- 匿名账号创建失败：提示"初始化失败，请重试"

### 业务规则
- 用户名必填
- 不要求唯一
- 禁止纯空格、超长、敏感违规内容

### 页面/交互说明
- 页面原型说明占位：`P-01 用户首次进入弹窗`
- 弹窗应为强引导，但不应营造注册感

### 状态变化
entry_init -> username_popup -> session_initializing

### 优先级
P0

---

## 8.2 会话自动初始化流程

### 功能目标
完成会话上下文建立，为首轮聊天做准备。

### 触发条件
用户名提交成功或已有匿名身份进入页面。

### 前置条件
匿名身份可用。

### 主流程
1. 前端请求创建会话。 
2. 服务端返回会话 ID。 
3. 前端建立 WebSocket。 
4. 建联成功后，机器人准备发送首问。 
5. 页面可进入正常聊天状态。 

### 分支流程
- 已有最近未过期会话：
 - 可恢复历史上下文或创建新会话，Demo 阶段建议默认创建新会话并仅保留简要上下文能力。

### 异常流程
- 创建会话失败：提示刷新/重试
- WebSocket 建联失败：自动重连，失败后手动重试

### 业务规则
- 会话创建必须在用户名初始化之后
- 会话未建立前不得发送业务消息
- 建联成功后才能发送首问

### 状态变化
session_initializing -> ws_connecting -> first_greeting_ready -> chat_ready

### 优先级
P0

---

## 8.3 机器人主动首问流程

### 功能目标
以自然方式让用户进入表达状态。

### 触发条件
会话创建成功且 WebSocket 已连接。

### 前置条件
聊天页面已准备完成。

### 主流程
1. 机器人发送欢迎语。 
2. 欢迎语中包含用户名和可咨询范围提示。 
3. 若配置了快捷问题，则同时展示快捷问题区。 
4. 用户开始自由输入。 

### 分支流程
- 若用户曾有历史会话摘要：
 - 可展示"继续咨询"类轻提示，P1。

### 异常流程
- 首问发送失败：
 - 重试一次
 - 失败则展示系统提示"连接已恢复，你可以直接输入问题"

### 业务规则
- 首问不得连续要求问题、诉求、订单号三类信息同时提供
- 首问应包含"可直接说问题"的引导信息

### 页面/交互说明
- 页面原型说明占位：`P-02 聊天页首问状态`
- 可配置首问模板与快捷问题

### 优先级
P0

---

## 8.4 用户自由表达 + 系统追问补全流程

### 功能目标
基于自然语言输入，决定直接处理还是补槽追问。

### 触发条件
用户发送业务消息。

### 前置条件
聊天状态正常。

### 主流程
1. 用户发送消息。 
2. 系统进入理解处理。 
3. 识别问题、诉求、订单号、场景类型。 
4. 判断信息是否足够： 
 - 足够：进入具体链路 
 - 不足：进入追问 
5. 追问后继续识别并处理。 

### 分支流程
- 用户一句话中已包含：
 - 问题 + 诉求 + 订单号 → 直接进入处理链路
- 用户仅表达情绪/模糊问题：
 - 发起场景澄清追问

### 异常流程
- 多轮仍识别不清：
 - 给出场景提示
 - 最终进入工单兜底

### 业务规则
- 追问一次只补一个关键字段
- 已有字段不得重复追问
- 若当前问题属于售前知识类，不得追问订单号

### AI 处理策略
- 优先做联合识别
- 识别置信度低时不直接执行，只追问
- 追问话术需自然、短句、单目标

### 状态变化
chat_ready -> bot_processing -> waiting_user_reply / completed

### 优先级
P0

---

## 8.5 售前咨询流程

### 功能目标
完成商品、运费、发货、服务说明等售前咨询自动答复。

### 触发条件
系统识别为售前咨询主意图。

### 前置条件
用户问题不依赖明确售后订单上下文。

### 主流程
1. 识别为售前咨询。 
2. 判断是否属于纯知识答疑。 
3. 检索知识库。 
4. AI 进行客服化组织。 
5. 回复用户。 
6. 可附带相关问题建议。 

### 分支流程
- 如果问题模糊：
 - 追问具体场景，如"你想了解运费、发货还是售后说明呢？"
- 如果知识可部分命中：
 - 回答已知部分 + 提示补充

### 异常流程
- 知识未命中：
 - 先尝试追问
 - 再提供保守解释
 - 必要时工单

### 业务规则
- 不强制订单号
- 售前问题默认不触发业务工具
- 不承诺库存、发货等动态信息，除非系统有可靠数据来源

### 优先级
P0

---

## 8.6 售后咨询流程

### 功能目标
处理订单、物流、售后状态类咨询。

### 触发条件
识别为售后咨询或订单/物流相关问题。

### 前置条件
用户问题与具体订单可能相关。

### 主流程
1. 识别为售后咨询。 
2. 判断是否需要订单号。 
3. 若需要则补问订单号。 
4. 获取订单号后进入查询/解释链路。 
5. 回复当前处理结果。 

### 分支流程
- 用户主动提供订单号：
 - 直接进入查询/解释
- 用户拒绝提供订单号：
 - 提供一般性规则说明，不进入具体判断

### 异常流程
- 订单号格式异常：
 - 提示重新提供
- 查询异常：
 - 道歉说明 + 工单引导

### 业务规则
- 订单相关场景中，订单号通常必填
- 若用户不提供订单号，只能提供通用说明，不能给出个单结论

### 优先级
P0

---

## 8.7 退款咨询类流程

### 功能目标
回答"能不能退、怎么退、多久到账、退款规则是什么"等咨询问题。

### 触发条件
识别为退款咨询类意图。

### 前置条件
用户尚未明确要求发起退款动作，或语义存在明显咨询倾向。

### 主流程
1. 识别为退款咨询。 
2. 判断是否为纯规则咨询还是个单资格咨询。 
3. 纯规则咨询： 
 - 知识检索 
 - AI 润色回复 
4. 个单资格咨询： 
 - 补问订单号 
 - 资格查询或规则解释 
5. 回复结果。 

### 分支流程
- 咨询中带有较强执行倾向但不够明确：
 - 先解释当前规则，再确认是否需要继续为其申请退款

### 异常流程
- 资格判断失败：
 - 不进入执行
 - 转为说明 + 工单建议

### 业务规则细化
- 以下表达优先判为咨询：
 - "能不能退"
 - "怎么退"
 - "可以提现吗"
 - "多久到账"
 - "退款规则是什么"
- 出现"退款"关键词但无动作词时，不默认进入执行链路
- 资格查询可辅助解释，但不代表自动申请

### AI 处理策略
- 先判断是否在问"规则 / 时效 / 能否"
- 如果用户原话更像"求结论"，但没有明确动作词，先咨询后确认，不直接执行

### 优先级
P0

---

## 8.8 退款执行类流程

### 功能目标
在条件满足时完成退款申请。

### 触发条件
识别为退款执行类意图。

### 前置条件
用户明确表达执行诉求。

### 主流程
1. 识别为退款执行。 
2. 判断订单号、必要参数是否完整。 
3. 参数不足时补问。 
4. 参数完整后做资格校验。 
5. 校验通过后调用退款工具。 
6. 返回工具结果。 
7. AI 组织解释性回复。 

### 分支流程
- 用户同时包含"问题 + 执行诉求"：
 - 直接识别进入执行准备
- 用户表达含糊：
 - 先确认是否要发起退款

### 异常流程
- 参数不足多轮仍未补齐：
 - 无法执行，提示补充或工单
- 校验不通过：
 - 输出原因解释，不触发工具
- 工具异常：
 - 说明未完成 + 工单引导

### 业务规则细化
- 以下表达优先判为执行：
 - "帮我退款"
 - "我要申请退款"
 - "发起退款"
 - "这个订单给我退了"
- 工具触发前必须满足：
 - 订单号有效
 - 用户执行意图明确
 - 资格校验通过
- 系统不得因模型猜测自动发起退款

### AI 处理策略
- 执行类回复结构建议：
 1. 说明正在处理/已处理
 2. 说明处理结果
 3. 说明后续时效或建议

### 优先级
P0

---

## 8.9 纯知识答疑流程

### 功能目标
完成对规则、说明、时效等文本类问题的自动回答。

### 触发条件
识别为纯知识答疑。

### 前置条件
问题可由知识库直接支持。

### 主流程
1. 知识检索 
2. 候选结果筛选 
3. AI 客服化组织 
4. 结果返回 

### 分支流程
- 知识部分命中：
 - 回答命中部分 + 追问缺失条件
- 用户继续追问：
 - 结合上下文重新检索或解释

### 异常流程
- 无结果：
 - 不直接说查不到
 - 进入未命中兜底

### 业务规则
- 不直接展示原始文档全文
- 回答要保留规则准确性，不可杜撰

### 优先级
P0

---

## 8.10 规则解释答疑流程

### 功能目标
针对"为什么不行/为什么失败/为什么不能退"等场景给出解释。

### 触发条件
用户表达中存在明确解释诉求。

### 前置条件
当前问题需要结合规则和状态理解。

### 主流程
1. 识别解释型问题。 
2. 判断是否需订单号或状态上下文。 
3. 若需要则补问。 
4. 获取规则依据与业务状态。 
5. AI 生成解释性话术。 
6. 返回原因 + 建议。 

### 分支流程
- 若仅有规则没有状态：
 - 给通用解释
- 若有状态无明确规则文案：
 - 给保守解释 + 引导进一步处理

### 异常流程
- 状态获取失败：
 - 不硬解释
 - 给出保守回复 + 工单

### 业务规则细化
- 解释类回复必须包含：
 - 当前结论
 - 主要原因
 - 可操作建议
- 若原因不确定，不可强行下定论

### 优先级
P0

---

## 8.11 意图不明确追问流程

### 功能目标
将模糊输入收敛到可处理场景。

### 触发条件
当前消息意图置信度低或无法映射到主意图。

### 主流程
1. 识别不明确 
2. 发起一轮自然追问 
3. 若仍不清晰，给出轻量场景选项 
4. 若仍无法识别，工单兜底 

### 分支流程
- 用户补充清晰：进入对应主链路
- 用户继续情绪化但不提供信息：优先安抚后再次引导

### 异常流程
- 连续 3 轮无法识别：
 - 进入工单

### 业务规则
- 不允许连续多轮"你请详细描述"式空泛追问
- 第二轮起应给具体范围提示

### 优先级
P0

---

## 8.12 知识未命中兜底流程

### 主流程
1. 检索无高置信结果 
2. 判断是否可通过追问缩小范围 
3. 可缩小则追问 
4. 不可缩小则给保守解释 
5. 必要时工单 

### 业务规则
- 知识未命中不等于系统直接失败
- 未命中场景应尽量保留用户信任，不让用户感知系统"不会"

### 优先级
P0

---

## 8.13 工具失败兜底流程

### 主流程
1. 工具调用失败或超时 
2. 用户收到失败说明 
3. 提供"稍后重试"或"提交工单"路径 
4. 记录失败埋点 

### 业务规则
- 不得把系统异常包装成用户资格不满足
- 工具失败要与资格不通过区分清楚

### 优先级
P0

---

## 8.14 工单引导流程

### 功能目标
承接无法由机器人完成闭环的场景。

### 触发条件
- 连续识别失败
- 知识无结果
- 工具失败
- 问题超边界
- 用户不接受当前结果

### 主流程
1. 说明当前机器人无法直接完成处理 
2. 总结已收集信息 
3. 展示工单引导卡片 
4. 触发工单提交流程或模拟提交 

### 页面/交互说明
- 页面原型说明占位：`P-03 工单引导卡片`
- 工单卡片需显示：
 - 问题摘要
 - 已收集订单号（如有）
 - 下一步动作按钮

### 优先级
P0

## 10.4 WebSocket 消息通讯

### 功能目标
提供实时、低延迟、可恢复的聊天体验。

### 用户价值
像即时客服一样自然交流。

### 触发条件
会话已创建。

### 前置条件
网络可用。

### 主流程
1. 建立连接 
2. 发送/接收消息 
3. 心跳维护 
4. 异常断开后自动重连 
5. 恢复会话上下文 

### 分支流程
- 短时断开自动恢复
- 长时断开提示用户手动重试

### 异常流程
- 建联失败
- 发送失败
- 接收超时
- 重连失败

### 业务规则细化
- 消息需具备唯一 ID
- 同一消息不得重复渲染多次
- 断线重连后需保证当前会话不串线

### 页面/交互说明
- 顶部展示连接状态
- 发送失败消息需可见

### 状态变化
connected / reconnecting / disconnected

### 兜底策略
连接异常时允许页面保留当前消息，并提示重试。

### 优先级
P0

### 验收建议
- 断网恢复后会话可继续
- 消息不丢失、不重发多次显示

P0

## 10.6 多轮对话管理

### 功能目标
让用户在自然对话中逐步完成闭环。

### 业务规则细化
- 单轮结果未完成时，下一轮优先承接当前链路
- 若用户明确切换话题，可切换主意图
- 若用户在退款执行过程中忽然问到账时间，应允许局部答疑但不丢当前上下文

### AI 处理策略
- 保留当前任务上下文 + 支持局部插问
- 对新旧话题冲突时优先用户最新明确诉求

### 优先级
P0

## 10.7 问题与诉求联合识别

### 功能目标
在一轮中尽量识别"发生了什么"与"想做什么"。

### 业务规则细化
- 若同时识别成功，则不重复确认
- 若仅识别问题，系统可根据上下文推断当前更像咨询还是解释，但不做高风险执行
- 若仅识别诉求，需追问必要问题上下文

### AI 处理策略
输出建议包含：
- issue_type
- user_goal
- confidence
- need_followup
- missing_slots

P0

### 兜底策略
- 资格不明：补订单号
- 规则不清：保守解释

## 10.11 退款执行处理

### 业务规则细化
- 工具调用前必须有明确动作意图
- 用户说"看看能不能退"不能直接执行
- 工具返回成功不代表最终到账完成，只代表申请提交成功或处理结果明确

### 结果消息类型
- 已提交
- 当前不可提交
- 参数不足
- 系统失败

### 优先级
P0

## 10.12 文本知识库查询

### 业务规则细化

- 不适合用来回答动态单订单状态问题
- 命中结果应保留原始依据，便于调试与评估，但用户侧不直接展示原始内容

### 优先级
P0

## 10.13 规则解释型答疑

### 业务规则细化
- 回答中必须尽量体现"原因 - 依据 - 建议"三段结构
- 没有依据时不能强行解释
- 如果只有工具失败结果，则解释应归因为系统原因，不归因为用户

P0

- 避免过度道歉、空泛安抚、长篇套话

P0

## 11. AI 产品专项设计

## 11.1 大模型在本产品中的角色定位

| 能力 | 是否由大模型直接负责 | 说明 |
|---|---|---|
| 问题理解 | 是 | 主责 |
| 诉求识别 | 是 | 主责 |
| 实体抽取 | 是 | 主责 |
| 追问生成 | 是 | 主责 |
| 知识命中 | 否 | 由检索系统负责，模型做组织 |
| 退款资格判定 | 否 | 由规则/工具负责 |
| 执行结果判定 | 否 | 以工具结果为准 |
| 用户话术生成 | 是 | 主责 |

## 11.2 多轮对话策略

### 细化规则
1. 先承接当前问题，不跳回重新分类。 
2. 当前未完成链路优先级高于新一轮泛化推荐。 
3. 用户中途变更诉求时，应允许打断并切换。 
4. 多轮追问超过阈值仍未收敛时，进入工单。 

### 建议阈值
- 单一链路补槽追问建议不超过 2 轮
- 连续意图不清建议不超过 3 轮

## 11.3 用户自由表达后的理解策略

### 处理框架
每轮输入后，系统需判断：
1. 是否已识别主问题 
2. 是否已识别用户诉求 
3. 是否具备当前链路最小必要参数 
4. 是否能直接回答 
5. 是否需要追问 
6. 是否需要进入兜底 

## 11.4 问题 / 诉求联合识别策略

### 识别优先级
1. 明确动作意图 
2. 明确解释意图 
3. 明确咨询意图 
4. 模糊描述待追问 

### 产品约束
- 联合识别成功时，不做二次拆问
- 联合识别失败时，只补缺失项，不整体重问

## 11.5 订单号等实体参数抽取策略

### 必识别字段建议
- order_id
- refund_action_flag
- consult_flag
- explain_flag
- issue_desc

### 补槽策略
- 缺 order_id 且当前链路必需时追问
- 缺退款原因但执行不强依赖时可不追问
- 缺问题描述且当前为模糊输入时优先追问问题描述

## 11.6 参数缺失时的补槽追问策略

### 追问生成规则
- 简洁
- 单目标
- 有上下文承接
- 不像表单

### 追问示例
- "我先帮你看下，请把订单号发给我。"
- "你是想先了解退款规则，还是现在就申请退款？"
- "你说的'这个不行'，是指物流、退款还是售后处理呢？"

## 11.7 主意图归类与动态分流策略

### 分流决策规则
- 主意图 + 当前轮槽位 + 会话历史 + 规则判断 决定最终链路
- 主意图只是入口，槽位和规则决定下一步动作

### 决策示例
- 主意图=退款；无执行动词；有"为什么" → 规则解释链路
- 主意图=退款；有执行动词；缺订单号 → 执行链路补槽
- 主意图=退款；无订单号；有"多久到账" → 纯知识答疑

## 11.8 退款咨询 vs 退款执行的区分策略

| 语言特征 | 默认判断 |
|---|---|
| 能不能退、可以提现吗、多久到账 | 咨询 |
| 帮我退、我要退款、发起退款 | 执行 |
| 为什么不能退 | 解释 |
| 我要退款多久到账 | 执行优先，执行结果后补充时效说明 |

## 11.9 纯知识答疑策略

### 使用条件
- 问题不依赖个单动态状态
- 知识库可支撑
- 不涉及高风险动作

### 结果要求
- 语言客服化
- 长度适中
- 不罗列原文

## 11.10 规则解释答疑策略

### 使用条件
- 用户在问原因
- 问题依赖状态 + 规则
- 用户希望理解"为什么"

### 结果要求
- 先结论
- 再原因
- 最后建议

## 11.11 AI 润色回复策略

### 文案约束
- 语气专业、友好、简洁
- 不油腻、不夸张、不承诺超权限
- 不要"系统提示式"生硬文案

## 11.12 AI 回复安全边界与约束策略

### 必须禁止
- 伪造退款成功
- 伪造规则依据
- 伪造人工将联系
- 伪造订单状态

### 必须保守
- 不确定时说明需要更多信息
- 工具失败时坦诚说明

## 11.13 模型不确定时的保守回复策略

### 建议模板
- "我先根据你现在提供的信息做一个初步判断……"
- "这个我还不能直接确认，你把订单号发我，我继续帮你看。"
- "目前还不能直接给出确定结果，我建议为你记录跟进请求。"

## 11.14 AI 失败兜底策略

### 兜底层次
1. 追问补充 
2. 规则解释 
3. 建议路径 
4. 工单

## 12. 关键业务规则

### 12.1 用户名采集规则
- 必填
- 2~20 字
- 允许重复
- 不允许空格提交
- 不允许敏感/违规内容

### 12.2 匿名账号生成规则
- 用户名提交成功后生成
- 生成后写入本地标识
- 匿名账号用于会话归属与工单补充

### 12.3 会话创建触发规则
- 匿名身份 ready 后触发
- 用户名未提交不得创建
- 重复点击提交需防重

### 12.4 首次问候触发规则
- 会话创建成功 + WebSocket connected 后触发
- 只在进入聊天时自动触发一次

### 12.5 售前订单号非必填规则
- 运费、发货、规则说明等不要求订单号
- 除非用户问题明确依赖个单状态

### 12.6 订单号必填场景规则
- 订单/物流问题
- 退款执行问题
- 规则解释且依赖当前订单状态
- 当前资格判断问题

### 12.7 问题/诉求联合识别规则
- 同轮优先同时识别
- 缺一补一，不重问全量
- 有执行意图时优先识别是否进入执行链路

### 12.8 退款咨询与退款执行区分规则
- 咨询词：能不能、怎么、多久、规则、为什么
- 执行动词：申请、发起、帮我退、我要退
- 有歧义先咨询后确认

### 12.9 退款工具触发规则
必须同时满足：
1. 当前主链路为退款执行 
2. 订单号完整有效 
3. 资格校验通过 
4. 未命中系统风控/异常拦截 
5. 工具服务可用 

### 12.10 可退款资格判断规则
建议最小规则集：
- 订单存在
- 订单状态允许退款
- 未超过售后有效期
- 未重复发起相同申请
- 无冲突中的售后流程

### 12.11 知识库查询触发规则
- 纯知识答疑默认触发
- 规则解释可联合触发
- 工具结果说明可引用规则知识

### 12.12 规则解释型问答触发规则
触发表达包括但不限于：
- 为什么不能退款
- 为什么失败
- 为什么不行
- 怎么还是不可以

### 12.13 知识无结果处理规则
1. 先追问缩小范围 
2. 再给保守解释 
3. 最终工单兜底 

### 12.14 参数缺失追问规则
- 每次最多追一个关键参数
- 订单号优先级高于其他字段
- 追问 2 轮未补齐则考虑兜底

### 12.15 用户连续模糊表达处理规则
- 第 1 次：自然追问
- 第 2 次：给场景选项
- 第 3 次：工单引导

### 12.16 工具失败处理规则
- 明确是系统问题而非业务不满足
- 提供稍后重试和工单路径
- 埋点记录失败原因

### 12.17 工单兜底触发规则
- 识别失败阈值达到
- 知识未命中无法收敛
- 工具失败
- 用户拒绝当前解释
- 当前问题超出能力边界

## 14. 异常与兜底设计

### 14.1 用户未填写用户名
- 用户可见：提交按钮后提示
- 系统处理：不创建匿名用户、不创建会话

### 14.2 用户名无效
- 用户可见：明确原因，不使用笼统"输入错误"
- 系统处理：停留当前弹窗

### 14.3 WebSocket 连接失败
- 用户可见：提示"连接中断，正在尝试恢复"
- 系统处理：自动重连，保留本地消息队列

### 14.4 WebSocket 中断与重连
- 用户可见：顶部状态提示
- 系统处理：恢复后拉齐未完成状态

### 14.5 会话初始化失败
- 用户可见：提示重试
- 系统处理：清理半初始化状态

### 14.6 意图无法判定
- 用户可见：自然追问
- 系统处理：保留当前原始输入用于工单摘要

### 14.7 问题与诉求不清晰
- 用户可见：补充方向性问题
- 系统处理：不直接进入执行

### 14.8 订单号缺失
- 用户可见：说明为什么需要订单号
- 系统处理：槽位状态标记为 required_missing

### 14.9 知识库无结果
- 用户可见：不暴露"检索失败"
- 系统处理：切换到兜底链路

### 14.10 工具返回失败
- 用户可见：系统暂时处理失败
- 系统处理：记录 request_id 与失败原因

### 14.11 退款不满足条件
- 用户可见：结论 + 原因 + 下一步
- 系统处理：标记 tool_not_executable

### 14.12 用户不接受解释结果
- 用户可见：提供进一步解释或工单路径
- 系统处理：标记 `user_not_accept`，提高兜底优先级

### 14.13 用户连续追问超出知识边界
- 用户可见：明确边界但不生硬
- 系统处理：引导人工后续处理

### 14.14 敏感/违规/越权问题处理
- 用户可见：礼貌拒绝并引导回客服场景
- 系统处理：安全拦截

## 18. 验收标准

> 本章将初稿"验收标准"进一步细化为"验收用例 / 示例级测试场景"。

### 18.1 功能验收

#### 用例 1：首次进入成功
- 前置条件：用户首次访问，无本地匿名身份
- When：用户进入 H5
- Then：展示用户名弹窗
- And：输入合法用户名后可进入聊天页并收到首问

#### 用例 2：已有匿名身份进入
- 前置条件：本地存在有效匿名身份
- When：用户再次访问 H5
- Then：不展示用户名弹窗
- And：直接进入会话初始化流程

### 18.2 交互验收

#### 用例 3：首问不机械
- 前置条件：会话创建成功
- When：机器人发送首问
- Then：首问不得同时要求问题、诉求、订单号
- And：用户能直接自由输入

#### 用例 4：追问不过度
- 前置条件：用户输入"这个订单怎么还不能退款"
- When：系统识别缺订单号
- Then：仅追问订单号
- And：不再同时询问问题类型和诉求类型

### 18.3 AI 识别效果验收

#### 用例 5：退款咨询识别
- 前置条件：正常聊天状态
- When：用户输入"退款多久到账"
- Then：系统识别为退款咨询
- And：走知识答疑 / 规则说明链路
- And：不触发退款工具

#### 用例 6：退款执行识别

#### 用例 7：规则解释识别
- When：用户输入"为什么这个订单不能退款"
- Then：系统识别为规则解释问题
- And：补问订单号后进入解释链路

### 18.4 知识命中效果验收

#### 用例 8：纯知识问答命中
- When：用户输入"运费怎么收"

### 18.5 工具执行链路验收

- 前置条件：订单允许退款
- When：用户输入"帮我申请退款"，并提供合法订单号

- And：返回"已提交"结果卡片

- And：返回不可执行原因解释

- When：用户发起退款

- And：提供稍后重试或工单路径

## 19. 风险与待确认事项

### 19.1 业务规则风险
- 退款资格规则边界不清将直接影响执行链路可靠性
- 规则解释话术若与业务真实规则不一致，风险高

### 19.2 AI 识别准确率风险
- 退款咨询 / 执行 / 解释三类语义高度接近
- 多轮上下文切换可能导致误分流

### 19.3 知识库数据准备风险
- 规则条目不成体系会影响回答质量
- 无法支撑解释型答疑时会过度依赖保守回复

### 19.4 工具依赖风险
- 接口可用性、幂等性、错误码规范性都会影响产品体验

### 19.5 WebSocket 稳定性风险
- 断线重连、消息重复、会话串线需重点关注

### 19.6 Demo 与正式产品边界风险
- Demo 中若 Mock 太多，需对评审方明确边界
- 若首版逻辑写死，将影响后续扩展

### 19.7 待确认问题列表
1. Demo 阶段是否支持恢复最近一次会话。 
2. 订单号格式是否已有稳定规则。 
3. 退款资格判断是否有统一规则源。 
4. 工具接口是否返回统一结果枚举。 
5. 工单链路是模拟还是接真实系统。 
6. 快捷问题是否支持后台配置。

## 20. 研发实现关注点（新增）

### 20.1 状态管理复杂点
- 用户状态、会话状态、连接状态、轮次状态、工具状态并存，需避免相互覆盖。
- "处理中""等待补充""工具中""异常中"需要清晰状态机，避免前端展示错乱。

### 20.2 WebSocket 会话一致性风险
- 重连后消息顺序、重复消息、延迟消息处理需要有一致性策略。
- 需要明确前端展示状态与服务端权威状态的同步方式。

### 20.3 AI 结果不确定性风险
- 模型输出可能不稳定，必须通过规则约束高风险节点。
- 研发需明确哪些字段可由模型直接驱动，哪些必须由规则/工具确认。

### 20.4 规则判断与模型判断边界
- 模型负责"理解"和"建议路径"
- 规则负责"是否可执行"
- 工具负责"执行结果"
- 这三层边界必须在实现层保持稳定，不能混用

### 20.5 工具调用前后的状态收敛问题
- 调用前：必须参数完整
- 调用中：页面要有处理中提示
- 调用后：要收敛为明确结果状态，而不是悬空

### 20.6 消息类型与渲染分离
- 不同消息类型建议前后端协议中显式区分，避免前端靠文案猜类型
- 特别是工具结果卡、工单卡、系统状态消息

### 20.7 兜底链路一致性
- 知识未命中、工具失败、识别不清都可能进入兜底，但原因不同，埋点和前端展示需区分

### 20.8 可观测性
- 关键链路需有 trace/request_id
- AI 识别结果、知识命中结果、工具调用结果最好可关联，便于问题排查

## 21. 测试关注点（新增）

### 21.1 首轮交互体验
- 首问是否自然
- 是否无过度盘问
- 用户进入是否顺畅

### 21.2 追问是否过度
- 是否在不必要场景追问订单号
- 是否重复追问已知信息
- 是否每轮只补一个关键字段

### 21.3 订单号缺失分支
- 售前是否不强制要订单号
- 售后是否能正确补问订单号
- 非法订单号是否正确拦截

### 21.4 知识未命中兜底
- 是否会直接回复"查不到"
- 是否能给合理的保守解释
- 是否能正确进入工单兜底

### 21.5 退款咨询与执行区分
- "能不能退"是否不会误执行
- "帮我退款"是否能正确进入执行补槽
- "为什么不能退款"是否走解释链路

### 21.6 工具失败兜底
- 工具异常时是否明确说明
- 是否提供后续路径
- 是否与"不可退款"区分清楚

### 21.7 WebSocket 异常恢复
